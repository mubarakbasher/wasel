## Payment flow fix ŌĆö DONE (on `dev` `9a9b38a`, pending staging)
Two defects in the manual bank-transfer flow, fixed backend + admin. **No migration** ŌĆö `payments.receipt_url` was already nullable; this is a query/visibility + UI change.
- **Receipt-less payments no longer reach the admin.** The `payments` row is created up front at `POST /subscription/request` (so the bank `reference_code` can be shown) with `receipt_url=NULL`; receipt upload is a separate later call (`POST /subscription/receipt`). A user who viewed the bank details and backed out left a `pending`, receipt-less row the admin could see ŌĆö and approve, activating a subscription with no proof of payment. Fixes:
  - `getPayments()` always filters `p.receipt_url IS NOT NULL` (every tab).
  - `reviewPayment()` carries the **same guard on its `UPDATE ŌĆ” WHERE`**, so approve/reject are refused at the action site too, not just hidden in the list (defense-in-depth vs direct API calls ŌĆö security-auditor finding; list-hide alone was bypassable).
  - `getStats()` pending-payments tile now matches the actionable queue.
  - Also fixes the **"all" tab**, which showed only pending because the service defaulted `status || 'pending'`; the status predicate is now applied only when a status is provided.
- **Admin payment-detail view.** Backend `LEFT JOIN plans` for the display name (`plan_name`). Clicking a row opens a detail modal: payer, plan, amount, reference code, status/rejection reason, dates, and an **inline receipt-image preview** (via the host-allowlisted `resolveAssetUrl` ŌĆö confirmed no XSS/phishing surface); Approve/Reject available from the modal.
- Verified: backend `tsc` clean + **254 tests** (new `backend/src/tests/adminPayments.test.ts` covers the `getPayments` WHERE-clause + the `reviewPayment` receipt guard); admin `tsc -b && vite build` clean. Reviewed by security-auditor + code-reviewer ŌĆö both blocking findings fixed in the same commit.

## Payment recovery fix (mobile) ŌĆö DONE (on `dev`, pending staging)
Follow-up to `9a9b38a` above. That change (correctly) hid receipt-less `pending` payments from the admin, but it removed the *accidental* safety net ŌĆö the admin used to see and reject the orphan, which handed the user a `rejected` payment that had Cancel/Resubmit buttons. Operators then hit a trap: a user who backed out of the payment stepper before uploading was left with a `pending`, receipt-less payment that had **no** recovery buttons in the app (they rendered only for `rejected`), couldn't be uploaded to after an app restart (the `paymentId` lived only in in-memory `lastRequest`), and was invisible to the admin. **Mobile-only fix ŌĆö the backend already permits cancel + (re)upload from `pending`** (`subscription.service.ts:326-328,373-375`); no backend change, no migration.
- `mobile/lib/screens/settings/payments_screen.dart` ŌĆö the Cancel + Resubmit action row is no longer gated on `isRejected`; **any non-terminal payment** (pending or rejected) shows recovery actions: pending-no-receipt ŌåÆ "Upload receipt" + "Cancel", pending-with-receipt ŌåÆ "Replace receipt" + "Cancel" (+ an "awaiting review" note), rejected ŌåÆ unchanged. Reuses the existing `cancelPayment`/`resubmitReceipt` provider methods (they key off `PaymentRecord.id`, not `lastRequest`).
- `mobile/lib/screens/subscription/payment_screen.dart` ŌĆö `initState` also `loadPayments()`; `_handleUpload` falls back to the current pending payment's id when `lastRequest` is null (survives an app restart); a **`PopScope` back-guard** intercepts leaving the stepper with a pending, receipt-less payment ŌåÆ "Upload later / Cancel payment / Stay".
- `mobile/lib/models/payment_record.dart` ŌĆö new `hasReceipt` getter.
- i18n: new EN+AR keys in `app_localizations.dart` (`payments.uploadReceipt/replaceReceipt/pendingUploadHint/awaitingReviewNote`, `payment.leaveTitle/leaveBody/uploadLater/cancelPayment/stay`).

## Data-limit live enforcement ŌĆö DONE (on `dev`, 2026-07-19, pending staging)
Operator-reported: a **data**-limited voucher (e.g. 100 MB) kept working **past** the cap mid-session; it only failed on the *next* reconnect. Root-caused (3 Explore agents across backend/FreeRADIUS/the working Wassal reference + official MikroTik/FreeRADIUS docs). **No mobile change.**
- **Root cause:** a data voucher wrote only a `Max-Total-Octets` *check* attribute consumed by the `max_total_octets` **sqlcounter**, which runs **only in `authorize {}`** (Access-Request time). Nothing handed the router a byte ceiling it could enforce live, and no CoA fired on quota ŌĆö so the session ran uncapped until re-auth rejected it. (Time limits already work live because they map to a router-enforced `Session-Timeout` reply.)
- **Fix ŌĆö three layers:**
  1. **Live (router-enforced):** voucher creation now writes `Mikrotik-Total-Limit` (+ `Mikrotik-Total-Limit-Gigawords` for >4 GB) to `radreply` (`voucher.service.ts`; low/high-word split mirrors the existing `Max-Total-Octets` split). The `sql` module returns these on Access-Accept (attrs 17/18 already in `freeradius/dictionary`) so RouterOS disconnects at the cap ŌĆö **no FreeRADIUS config change needed for this layer.** Migration `037_mikrotik_total_limit_backfill.sql` backfills existing active data vouchers (idempotent `NOT EXISTS`, `limit_value >= 0` guard).
  2. **Backstop (server CoA):** new `jobs/dataUsageCoaDisconnect.ts` (mirrors `validityCoaDisconnect.ts`) ŌĆö every 30 s, force-disconnects active sessions of data vouchers whose **cumulative** `radacct` usage Ōēź limit (full-precision BIGINT, so >4 GB is correct). Guards: `isSafeAcctSessionId` before radclient, `acctstarttime` within 24 h + `LIMIT 200`, and an in-flight guard against overlapping ticks. Covers RouterOS versions that ignore `Mikrotik-Total-Limit` (a documented caveat) and the cross-session case.
  3. **Correctness (FR config ŌĆö the one FR change):** removed the 32-bit `max_total_octets` sqlcounter from `sites-enabled/default` `authorize` + its `mods-enabled/sqlcounter` block (kept `max_all_session`). rlm_sqlcounter's `check_name` is 32-bit, so it *mis-rejected* >4 GB vouchers early. Auth-time cumulative reject is now owned by the existing full-precision `usageLimitEnforcement` cron (latches `Auth-Type := Reject`).
- **ŌÜĀ’ĖÅ Known trade-off (validate on staging):** removing the sqlcounter drops the old *0-latency* Ōēż4 GB auth-time reject; a determined reconnect within the cron/CoA window (Ōēż30 s) can get one fresh session before the reject latches. The honest-user path is still stopped instantly by the router (Layer 1). This is inherent to the static-reply approach chosen to support >4 GB; the airtight follow-up (keep a 64-bit-safe auth-time check for Ōēż4 GB, or restrict the check attr to Ōēż4 GB so the sqlcounter can stay) is deferred pending whether staging shows the window is exploitable in practice.

## Gotchas already hit (don't re-debug these)
- **Postgres password Ōåö volume:** `POSTGRES_PASSWORD` only takes effect on the **first** init of the `postgres_data` volume. If it ever differs from `backend/.env` `DB_PASSWORD`, the backend dies with PG error `28P01`. Fix: make the two identical, then `docker compose down -v` to drop & re-init the volume.
- **Compose env-file:** every `docker compose` command needs `--env-file /etc/wasel/compose.env`, OR rely on the `/opt/wasel/.env` copy (gitignored) which Compose auto-loads.
- **Admin baked prod URL:** `admin/.env.production` hardcodes `VITE_API_URL=https://api.wa-sel.com/api/v1`. Before exposing `admin.wa-sel.cloud`, rebuild the admin with `admin/.env.production.local` (`VITE_API_URL=/api/v1`, served same-origin) or it will talk to **prod**. Verify the built bundle contains no `api.wa-sel.com`.
- **Terminal paste:** multi-line pastes can inject a `^[[200~` bracketed-paste marker that corrupts the first command (`docker: command not found`) ŌĆö run those commands one line at a time.
- **Backend secrets mount + dropped privileges:** the backend container's entrypoint drops to user `app` (uid 101, gid 102) even though `docker compose exec` enters as root. Files under the host-mounted `backend/secrets/` (ŌåÆ `/app/secrets` ro) must be readable by uid 101 ŌĆö a root-owned `700` dir makes Firebase init fail with a *misleading* `MODULE_NOT_FOUND` (Node maps the EACCES during resolution to "module not found"). Correct provisioning (done on staging 2026-07-16, repeat on prod at promote): drop the service-account JSON in `/opt/wasel/backend/secrets/firebase-service-account.json`, `chown -R 101:102 backend/secrets && chmod 700 backend/secrets && chmod 400 backend/secrets/*.json`, set `FIREBASE_SERVICE_ACCOUNT_PATH=/app/secrets/firebase-service-account.json` in `backend/.env`, recreate backend, confirm boot log "Firebase Admin SDK initialized".
- **Staging SMTP:** staging relays through **Resend** (`smtp.resend.com:465`, user `resend`, sending-scoped key ŌĆö 401 on `/domains` is normal). A MailHog catch-all also runs on the box (`127.0.0.1:1025`, UI `:8025`); pointing `SMTP_HOST=127.0.0.1 SMTP_PORT=1025` at it silently captures ALL mail locally ŌĆö that's what made Resend "go quiet" between Jun 30 and Jul 16 while `email_log` said `sent`.
